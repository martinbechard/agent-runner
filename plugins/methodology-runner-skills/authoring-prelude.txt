## MANDATORY: Authoring methodology — do NOT skip this

You MUST follow this process. Do NOT jump straight to writing a draft.
Do NOT skip variant testing. Do NOT present a single draft as final.
Every previous skill that skipped this process produced inferior results.

1. Propose 3-5 structural variants. Describe each in 2-3 sentences.
2. Draft ALL proposed variants as complete SKILL.md files. Do NOT
   pre-filter — prompt engineering is counter-intuitive and your
   intuition about which variant is "weaker" is often wrong.
   Write each variant to a SEPARATE file in /tmp/skill-variants-<skill-name>/
   (e.g., variant-a.md, variant-b.md). Do NOT try to produce all
   variants in one response — output length limits will force you
   to cut corners. One file per variant, full quality each.
3. Test ALL variants with claude -p using a NOVEL source (not the
   skill's own examples): both normal and adversarial pressure.
4. Present a comparison report. For each variant, list its strengths
   AND its specific weaknesses revealed by testing.
5. Build 1-2 hybrids that combine the winner's structure with fixes
   for its weak areas drawn from other variants. Use triple
   reinforcement (rule + example + counter-example) for any fix —
   don't propose single-layer alternatives when the methodology
   already proved triple is needed. Test the hybrids the same way.
6. Present final recommendation (hybrid vs. best pure). I pick or
   we iterate. Deploy when solid.
7. Before exiting: rm -rf /tmp/skill-variants-<skill-name>/ and any
   test workspaces you created under /tmp/. Leave no temp files.

If you find yourself writing a skill without having proposed variants
first, STOP. Go back to step 1.
